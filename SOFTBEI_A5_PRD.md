# 灵境导游 PRD 与 Codex 执行清单

## 1. 背景

中国软件杯 A5 赛题要求构建景区导览服务 AI 数字人。产品需要覆盖游客交互端和管理后台，具备多模态交互、智能问答、个性化路线推荐、知识库管理、数字人配置、游客感受度报告和数据大屏。

本项目基于工作区内的示范景区资料包开发：

- `示范景区公开资料包/灵山胜境 景点结构化数据集.docx`
- `示范景区公开资料包/灵山胜境：历史、文化、景点特色与个性化游览指南.docx`
- `示范景区公开资料包/景点景区旅游数据行为分析数据.xlsx`

## 2. 目标

### 2.1 比赛目标

冲击国一，优先做出高完成度、高可信度、高演示冲击力的作品。

### 2.2 产品目标

为游客提供可对话、可识景、可推荐路线、可避峰分流的 AI 数字人导游，并同时覆盖手机 App/PWA 与景区终端设备两个游客交互场景；为管理者提供知识维护、游客洞察、拥挤预警和运营数据分析能力。

### 2.3 非目标

- 不做真实支付/购票/订单。
- 不做真实景区室内高精度定位。
- 不训练自有大模型。
- 不做生产级 3D 数字人。

### 2.4 发布分层

P0 可演示纵向闭环：

- 手机端文本问答。
- 景点知识库解析。
- RAG 溯源回答。
- 基础数字人状态。
- 后台概览空数据/样例数据。

P1 比赛硬要求闭环：

- 语音输入/输出。
- 数字人口型和表情同步。
- 景区终端 `/kiosk`。
- 图片识景最小闭环。
- 路线推荐。
- 终端路线二维码带走。
- 交互日志和反馈。

P2 国一增强：

- 拥挤度感知路线分流。
- 路线推荐置信度和决策链。
- 低置信问题/知识缺口闭环。
- 管理员运营事件控制台。
- 准确率、识景率、路线约束和澄清追问评测面板。

P2 降级项：

- 完整知识库 CRUD。
- 数字人风格配置。
- 复杂情绪趋势。
- 深度行为数据行业洞察。
- 游记卡、方言讲解、原生 App 打包。

### 2.5 国一收敛原则

本项目不再按“功能清单”平均推进，而是围绕三条闭环交付：

1. 可信导览闭环：游客提问 -> RAG 检索 -> 来源卡片 -> 数字人语音讲解 -> 低置信澄清 -> 后台记录问题 -> 评测看板证明准确率。
2. 路线分流闭环：自然语言偏好 -> Route Planner 评分 -> 拥挤点错峰/替代 -> Kiosk 二维码带走 -> 手机逐站讲解。
3. 运营改进闭环：交互日志 -> 热门问题/低置信问题/反馈趋势 -> 知识缺口 -> 管理员补 FAQ -> 重建索引 -> 评测提升。

如果某个功能不能明显服务以上闭环，默认降级或不做。

## 3. 用户角色

### 3.1 游客

诉求：

- 想知道某个景点是什么、有什么故事、怎么游玩。
- 想根据自己的兴趣和时间得到路线。
- 希望避开人多排队的点位，或者在拥挤时获得替代路线。
- 希望像跟真人导游交流一样自然。
- 迷路、时间紧、带孩子等情况下需要即时帮助。

使用入口：

- 手机 App/PWA：适合游前规划、游中陪伴、拍照识景、扫码定位、路线收藏。
- 景区终端设备：适合游客中心/入口/热门点位的快速咨询、多人围观讲解、路线二维码带走。

### 3.2 景区管理员

诉求：

- 维护景区知识和讲解资料。
- 配置数字人形象、声音、讲解风格。
- 了解游客最关心什么、哪里不满意、哪些知识缺失。
- 了解热门点拥挤趋势，并通过路线推荐引导游客分流。
- 通过大屏查看运营情况。

### 3.3 评委

诉求：

- 快速看到赛题要求是否完整实现。
- 看到多模态大模型、知识库、数字人、后台分析的技术含量。
- 看到作品能否真实落地到景区。

## 4. 核心用户故事

1. 游客进入系统，数字人欢迎并询问游玩偏好。
2. 游客语音问：“九龙灌浴几点表演？”
3. 系统检索知识库，数字人回答演出时间、建议提前到场，并展示来源卡片。
4. 游客上传九龙灌浴照片，系统识别景点并自动讲解。
5. 游客说：“我带孩子，只有 4 小时，想轻松一点。”
6. 系统生成亲子轻松路线，包含每站停留时间和讲解重点。
7. 游客反馈“人太多，有点赶”，数字人安抚并推荐缩短路线。
8. 管理员在后台看到热门问题、低置信度问题、满意度和情绪趋势。

### 4.1 评分项对照

| 赛题要求 | 产品实现 | 演示位置 | 证明材料 |
|----------|----------|----------|----------|
| 游客交互端 | 手机 PWA + Kiosk 终端 | `/`、`/kiosk` | 页面演示、截图、视频 |
| 管理后台 | 运营概览、拥挤预警、知识缺口、评测指标 | `/admin` | 日志、图表、eval 报告 |

| 本地知识库 | docx/xlsx 解析、景点卡、知识切片、RAG | 问答来源卡片 | `data/processed`、sources |
| 事实准确率 | QA eval、来源约束、低置信澄清 | 评测看板 | eval reports |

### 4.2 7 分钟演示主线

演示只讲一条连贯故事：

1. 游客在手机端问景点问题，系统基于本地知识库回答并展示来源。
2. 游客上传景点图，系统给出识景候选，确认后进入讲解。
3. 游客说“带老人孩子，3 小时，别太挤，灵山大佛一定要去”，系统生成可解释路线。
4. 终端 Kiosk 生成同类路线，游客扫码在手机带走。
5. 管理后台展示拥挤预警、低置信问题、热门问题、反馈和评测指标。

## 5. 端形态与功能需求

### 5.1 手机 App/PWA 游客端

需求：

- 适配手机竖屏，支持添加到主屏幕。
- 展示数字人主区域、语音按钮、文本输入框、拍照识景入口、路线推荐入口。
- 支持拍照或上传图片识别景点。
- 支持路线收藏、继续讲解、扫码模拟当前位置。
- 支持弱网兜底：展示预置 FAQ、重点景点和路线模板。

验收：

- 手机浏览器可正常使用。
- 首页 2 秒内可交互。
- 核心按钮适合单手点击。
- 可完成语音问答、拍照识景、路线推荐和反馈。

### 5.2 景区终端 Kiosk 游客端

需求：

- 适配横屏触控大屏，提供 `/kiosk` 全屏模式。
- 闲时展示数字人欢迎页和热门问题。
- 按钮、文字、数字人尺寸适合 1 到 2 米外观看。
- 支持大按钮语音问答、文本快捷问题、路线推荐。
- 终端生成路线后展示二维码，游客可用手机扫码带走。
- 二维码带走的目标是把公共大屏的路线规划转移到游客个人手机，游客离开终端后仍能继续逐站讲解。
- 交互结束或超时后自动清空个人状态，回到欢迎页。

验收：

- 1920x1080 横屏下布局稳定。
- 可不依赖键盘完成主要流程。
- 超时 60 到 120 秒自动回到欢迎态。
- 路线二维码可被手机扫描打开移动端路线页。
- 分享码默认 30 分钟过期，避免公共终端泄露游客偏好。

### 5.3 共享游客端首页/主界面

需求：

- 展示数字人主区域。
- 展示语音按钮、文本输入框、拍照识景入口、路线推荐入口。
- 展示当前推荐景点或路线摘要。

验收：

- 页面打开后 2 秒内可交互。
- 数字人有欢迎动效。
- 移动端和桌面端都可用。

### 5.4 数字人问答

需求：

- 支持文本输入。
- 支持语音输入。
- 支持语音输出。
- 支持表情和口型同步。
- 回答需包含来源和置信度。
- 对低置信度问题优先澄清追问；仍无法确认时给出兜底话术。

验收：

- 能准确回答灵山大佛、九龙灌浴、灵山梵宫、五印坛城、祥符禅寺等重点问题。
- 回答中展示至少 1 条来源。
- TTS 播放时数字人口型变化。

### 5.4.1 自然语言回答准确率策略

目标：

- 不把 LLM 当作自由回答器，而是把自然语言输入拆成“意图理解、工具调用、证据约束、生成表达、自检追问”五步。
- 景点事实类回答必须绑定本地知识库来源；没有来源或来源分数过低时，不编造。
- 路线类回答必须调用受约束 Route Planner Agent；LLM 只能解析偏好和润色解释，不直接生成路线点位。

处理流程：

1. 意图分类：景点问答、路线推荐、路线重规划、讲解风格调整、识景追问、反馈投诉、未知问题。
2. 结构化解析：自然语言先转 JSON，例如 `theme`、`time_budget_minutes`、`group_type`、`intensity`、`avoid_crowd`、`style`、`operation`。
3. 受控工具调用：QA 调 RAG，路线调 Route Planner，重规划调 Route Memory Agent + Route Planner，讲解风格调 RAG + 风格改写。
4. 证据约束：事实、历史、演艺、参数类回答必须有 sources；sources 为空时转澄清或兜底。
5. 置信度自检：返回 `intent_confidence`、`retrieval_confidence`、`answer_confidence`、`needs_clarification`。

### 5.4.1.1 游客自然语言能力矩阵

目标：

- 游客不需要先选择“问答/推荐/拥挤/路线”功能，系统先做结构化理解，再分流到正确能力。
- “介绍、推荐、适合、怎么游览、看点、讲解”只是意图词，不是景区实体；不能因为这些词误触发 RAG。
- 能回答的常见导览需求不能过度拦截，例如“我对历史感兴趣，有什么景点推荐？”“介绍拈花湾”“现在人多吗？”。

能力矩阵：

| handler | 适用输入 | 约束 |
|---------|----------|------|
| `qa_rag` | 明确景点事实、文化、演艺、游览问答 | 必须返回本地 `sources`，无来源不编造 |
| `scenic_area_intro` | 介绍灵山胜境、介绍拈花湾、景区总览 | 基于本地 22 个景点分组生成摘要；范围不清时先澄清 |
| `interest_recommendation` | 历史、拍照、亲子、老人、自然、祈福、演出等兴趣推荐 | 基于景点标签、类别、简介规则评分，不接 LLM 自由推荐 |
| `comparison` | 两个景点或景区之间的适合度对比 | 目标不足时澄清 |
| `crowd_status` | 现在人多吗、哪里拥挤、临时关闭、演出提醒 | 只使用 mock 拥挤度和 manual_admin/mock_simulation 运营事件 |
| `route_planner` | 路线规划、时长、必去、避开、重规划 | 必须调用受约束 Route Planner |
| `clarification` | 指代、范围或对比对象不清 | 返回候选 chips |
| `out_of_scope` | 资料外问题、非景区主题 | 明确“不在本地景区知识库范围内”，sources 为空 |

验收：

- 新增 `eval_query_capability.py`，覆盖景区总览、兴趣推荐、对比、拥挤运营、RAG、路线、澄清和资料外兜底不少于 35 条样例。
- `/` 游客端按能力类型展示结构化结果，而不是把所有内容塞进同一段长文本。
- 后续 LLM 只能替换/增强结构化意图解析，不得绕过 RAG sources、Route Planner、Crowd/Operation 约束。

### 5.4.2 低置信度澄清追问

触发条件：

- 检索分数低或 sources 为空。
- 用户使用“这里”“那个喷水的地方”“大佛旁边”等模糊指代。
- 同时匹配多个景点或多个意图。
- 用户问题可能涉及票价、实时天气、酒店、交通管制等资料包外信息。

输出要求：

- 返回 `answer_type: clarification`。
- 给出一句清楚追问。
- 给出 2 到 4 个候选选项，例如“九龙灌浴”“五明桥香水海”“我想问附近路线”。
- 不把澄清追问计为失败回答；后台记录为低置信澄清事件。

### 5.4.3 智能讲解深浅调节

支持风格：

- 亲子版：更短、更生活化，避免复杂宗教术语。
- 历史深度版：强调历史、文化脉络和建筑/艺术细节。
- 30 秒短讲版：适合排队、赶路或短视频旁白。
- 拍照打卡版：强调取景、动线和注意事项。
- 安抚版：游客抱怨拥挤、疲劳或时间不足时使用。

约束：

- 风格只改变表达，不改变事实。
- 事实点必须来自 RAG sources、景点结构化字段或路线规划结果。
- 没有来源时只给一般游览建议，并明确“不确定”。

### 5.5 拍照识景

需求：

- 游客上传图片。
- 系统调用多模态模型识别景点。
- 识别后展示 Top 3 候选景点、置信度和确认按钮。
- 用户确认候选后展示景点卡片并自动讲解。
- 模型不可用时可使用样例图或手动选择降级。
- 首轮必须支持 mock provider，保证无 API key 时仍可演示。

验收：

- 上传成功后展示识别候选、置信度、确认入口和讲解内容。
- 至少准备 8 到 10 个演示样例。
- 失败时有清楚兜底，不阻塞演示。
- 不把低置信识景结果“一次判死”；置信度低或多候选接近时必须让游客确认。

### 5.6 个性化路线推荐

输入：

- 兴趣：历史文化、自然风光、亲子轻松、祈福朝圣、拍照打卡。
- 时长：2 小时、4 小时、6 小时、全天。
- 同行：独自、情侣、亲子、长辈、朋友。
- 体力：轻松、普通、深度。
- 当前位置：可由扫码点位、识景结果或手动选择景点模拟。
- 必去景点：用户明确要求必须保留的景点，例如灵山大佛、九龙灌浴。
- 可选景点：用户感兴趣但可被替代或延后的景点。
- 避开景点：用户明确不想去、已去过或不适合当前同行人群的景点。
- 拥挤度偏好：避开拥挤、可接受普通排队、不考虑拥挤。

候选池要求：

- 路线候选池必须来自全部已解析景点，而不是少数核心景点。
- 当前资料解析得到的 22 个景点都应支持作为必去、可选或避开约束。
- 经典 5 类路线模板只作为 seed，用于提供基础顺序和主题样例，不是候选上限。
- 非模板景点被用户设为必去时，系统必须尝试纳入路线并说明处理方式。
- 非模板景点被设为可选时，系统应提升其主题得分；如未纳入，需说明原因。

输出：

- 路线名称。
- 点位顺序。
- 每站预计停留时间。
- 每站讲解重点。
- 演艺时间提醒。
- 适合原因。
- 推荐置信度或综合评分。
- 每站评分拆解：主题匹配、时间匹配、体力适配、拥挤度影响。
- 每站约束类型：必去、系统推荐、替代推荐、错峰保留。
- 每站入选来源：必去约束、经典模板 seed、全量候选池、可选加权。
- 每站主题画像匹配说明：为什么该点适合亲子、历史、自然、祈福或拍照。
- 必去景点处理说明：立即前往、延后错峰、保留但提醒排队。
- 决策说明：为什么选择该路线、为什么跳过或延后某个拥挤点。

验收：

- 至少内置 5 条高质量路线模板。
- 全部已解析景点都能进入路线候选池。
- 路线规划以受约束评分器为主，LLM 只做解释和讲解润色，不直接自由决定路径。
- 生成路线可以一键进入逐站讲解。
- 支持“时间不够/太拥挤/太累/想多拍照”等场景下重新规划。
- 支持必去景点约束。必去景点不能因为拥挤被直接删除，只能调整顺序、延后或提示错峰。
- 在 mock 拥挤度数据下，高拥挤点会被降权或提示稍后返回。

景点路线画像：

- 每个景点应通过规则派生路线画像，包含 `family_score`、`history_score`、`nature_score`、`blessing_score`、`photo_score`、`route_priority`、`default_stay_minutes`、`is_core_landmark`。
- 画像可基于 `category`、`tags`、`name`、`summary/description`、`scenic_area` 生成。
- 画像分数用于补充全量候选池排序，不作为不可解释的黑盒模型。

### 5.6.1 拥挤度感知分流推荐

目标：

- 从“游客想怎么玩”升级为“游客怎样玩更舒服，景区怎样更均衡”。
- 当热门点拥挤时，系统能解释性地避开高峰点，推荐替代站点或调整顺序。
- 后台能看到拥挤点、分流建议和路线推荐偏好。

数据边界：

- 当前阶段使用 `mock_simulation` 拥挤度数据，不声称接入真实景区客流。
- 预留真实数据源字段，未来可接入闸机、预约、摄像头、Wi-Fi 探针、停车场或手工运营配置。

输出要求：

- 每个景点显示 `crowd_level`：low、medium、high。
- 每个景点显示 `crowd_score`：0 到 100。
- 可选显示 `wait_minutes`。
- 路线说明中明确“已避开某高拥挤点”或“建议稍后返回某点”。

### 5.6.2 自然语言路线推荐

目标：

- 游客不用先点主题、时长、体力，而是直接用口语描述需求。
- 系统将自然语言解析为结构化路线偏好，再调用现有 Route Planner Agent。

示例输入：

- “我带孩子和老人，下午只有 3 小时，想轻松点，别太挤。”
- “我想多拍照，最好能看一个演出，4 小时左右。”
- “第一次来灵山，想看最经典的几个点，但不要走太远。”
- “灵山大佛和九龙灌浴一定要去，其他你帮我安排，尽量别排队。”
- “香月花街一定要去，九龙灌浴避开。”
- “我想去拈花湾的五灯湖和梵天花海。”

解析输出：

- `intent`: `route_recommend`
- `theme`
- `time_budget_minutes`
- `group_type`
- `intensity`
- `interests`
- `must_visit_attraction_ids`
- `optional_attraction_ids`
- `avoid_attraction_ids`
- `avoid_crowd`
- `crowd_tolerance`
- `start_attraction_id`
- `intent_confidence`
- `needs_clarification`

验收：

- 无 API key 时使用规则/mock parser。
- 有 LLM provider 时可切换 LLM parser，但必须输出结构化 JSON。
- 解析失败或置信度低时进入澄清追问，而不是随意生成路线。
- 用户提到“一定要去/必须看/不能错过”时解析为必去景点约束。
- 必去景点当前拥挤时，系统应保留并延后或提示错峰，而不是直接替换掉。
- 景点名识别必须基于全部景点列表，不能只覆盖核心五点。
- 短名称或别名存在歧义时进入澄清追问。

### 5.6.3 Route Memory Agent 与多轮重规划

目标：

- 让游客感觉系统记得上下文，例如“太累了，缩短一点”“人太多，换一个”“从梵宫出发重新安排”。
- Agent 只维护记忆和解析操作，不能自由编造景点或路线。

记忆内容：

- 游客偏好：同行、时间、体力、兴趣、避拥挤偏好。
- 景点约束：必去景点、可选景点、避开景点及其来源，例如用户明确提出、识景结果、上一次路线保留。
- 路线状态：当前 route_id、当前站点、已选站点、删掉/延后站点、高拥挤站点。
- 对话状态：最近请求、最近操作、调整原因、turn_count。

支持操作：

- `shorten`：缩短路线。
- `avoid_crowd`：避开拥挤点。
- `less_walking`：少走路，适合老人/疲劳。
- `more_photo`：增加拍照打卡点。
- `more_history`：增加历史文化讲解点。
- `start_here`：从当前景点重新规划。
- `set_must_visit`：设置或追加必去景点。
- `remove_must_visit`：用户明确放弃某个必去点时移除约束。

API 草案：

- `POST /api/routes/conversation`

输入：

- `session_id`
- `message`
- `current_route_id`

输出：

- `intent`
- `operation`
- `memory`
- `route`
- `reply`
- `confidence`
- `needs_clarification`

验收：

- 不使用真实账号，游客端可生成本地 mock session；Kiosk 匿名 session 超时清空。
- “太长了/时间不够”能触发缩短路线。
- “人太多/别排队”能触发避拥挤重规划。
- “老人累了/少走点”能触发少步行策略。
- “灵山大佛一定要去”能写入必去景点记忆。
- 必去点拥挤时，结果必须解释“保留但错峰/延后”的取舍。
- 结果必须解释保留、删除或延后哪些站点。

### 5.6.4 运营事件控制台

目标：

- 让景区管理者可以人工发布临时运营事件，作为真实客流硬件不可得时的可落地替代方案。
- 路线规划器读取运营事件后自动调整路线，并在游客端解释原因。

事件类型：

- `crowd`: 人流拥挤，例如“九龙灌浴当前拥挤，等待约 30 分钟”。
- `closed`: 临时关闭或维护，例如“灵山梵宫临时维护，建议绕行”。
- `show`: 演出提醒，例如“吉祥颂即将开始，建议提前 15 分钟到达”。
- `recommendation`: 运营推荐，例如“亲子游客推荐香月花街和梵天花海”。

规则：

- 运营事件优先级高于普通 mock 拥挤度，但低于用户明确必去约束。
- `closed` 事件命中非必去点时应移除或替换；命中必去点时必须提示风险并询问是否仍保留。
- `show` 事件可以提高相关景点得分或调整顺序。
- 所有事件都要标注 `source: manual_admin` 或 `mock_simulation`。

验收：

- 管理后台可新增、启用、停用运营事件。
- 游客端路线结果能展示“因运营事件调整”的 decision_trace。
- 不声称事件来自真实硬件数据，除非未来明确接入。

### 5.7 游客反馈

需求：

- 游客可提交满意度 1 到 5 分。
- 可选择标签：讲解清楚、路线合理、人多拥挤、信息不准、体验惊喜、还想了解。
- 可输入自由文本。
- 后端记录到交互日志。

验收：

- 后台可看到反馈趋势和原始反馈。

### 5.7.1 公共终端隐私

需求：

- Kiosk 端默认匿名会话。
- 超时后自动清空当前问答、路线偏好和识别结果。
- 不在终端展示上一位游客的历史问题。
- 管理后台只看聚合分析和脱敏日志。

验收：

- 终端空闲超时后回到欢迎页。
- 刷新 `/kiosk` 不恢复上一位游客的私人状态。
- 日志中不保存手机号、身份证等敏感字段。

### 5.8 管理后台概览

需求：

- 展示今日/本周服务人次。
- 展示热门问答。
- 展示游客满意度趋势。
- 展示情绪趋势。
- 展示平均响应延迟。
- 展示低置信度问题。
- 展示拥挤点预警和管理员运营事件。
- 展示评测通过率摘要：QA、识景、路线、约束、澄清、延迟。

验收：

- 图表来自真实交互日志或公开数据处理结果。
- 支持空数据时展示演示样例数据。
- 首屏优先展示服务人次、平均响应延迟、低置信问题数量、热门问题 Top 10、拥挤点预警和知识缺口待处理列表。

### 5.9 知识库管理

需求：

- 展示已解析文档。
- 展示景点卡片列表。
- 展示知识切片列表。
- 支持重新构建索引。
- 支持新增 FAQ。
- 展示知识缺口列表：低置信问题、无来源问题、游客反馈信息不准的问题。
- 支持从低置信问题生成 FAQ 草稿。
- 支持把知识缺口加入评测集。

验收：

- 管理员能看到每个景点的核心字段。
- 重新构建索引后游客端问答使用新内容。
- FAQ 草稿保存后能被重新索引并进入后续评测。

### 5.9.1 知识缺口闭环

流程：

1. 游客问题低置信、sources 为空、或被用户反馈“信息不准”。
2. 后端记录为 `knowledge_gap`。
3. 管理后台展示问题、匹配来源、置信度、建议 FAQ 草稿和状态。
4. 管理员标记为资料包外、忽略、补充 FAQ 或加入评测集。
5. 重新构建索引后，相关 eval 自动验证是否变准。

验收：

- 后台能看到知识缺口状态：open、drafted、resolved、ignored。
- 每个知识缺口都能追溯到原始 query 和触发原因。
- 加入评测集后能在评测看板看到结果变化。

### 5.10 数字人配置

需求：

- 配置数字人名称。
- 配置讲解风格：亲切、专业、活泼、禅意。
- 配置声音 provider。
- 配置默认表情和欢迎语。

验收：

- 配置保存后游客端生效。

### 5.11 游客行为数据分析

需求：

- 读取 xlsx 的 14 万行公开行为数据。
- 展示年龄、性别、景区类型、消费、停留、满意度分布。
- 生成行业洞察文字。

重要边界：

- 页面必须标注为“公开样例行为数据/行业画像”，不能声称是灵山真实运营数据。

验收：

- 至少 4 个图表。
- 至少 3 条自动生成的运营建议。

### 5.12 评测看板

目标：

- 把“可信度”从口头承诺变成可演示证据。
- 评委能在后台看到最新评测时间、样本数、通过率、失败样例和延迟。

指标：

- QA 准确率。
- 引用命中率。
- 幻觉拦截率。
- 低置信澄清准确率。
- 识景候选 Top1/Top3 成功率。
- 路线推荐通过率。
- 必去景点保留率。
- 拥挤点错峰解释率。
- 平均延迟和 P95 延迟。

验收：

- 后台能读取 `evals/reports/*_latest.json` 的摘要。
- 每项指标显示样本数、通过数、失败数、更新时间。
- 失败样例可展开查看，但不要求在首版支持复杂筛选。

## 6. UI/UX 设计系统

### 6.1 设计定位

产品气质：安静、可信、文化感、工具感。它是景区服务系统，不是营销落地页。

界面应该像“游客中心里的专业导览设备 + 手机上的贴身导游”，避免花哨包装。视觉重点放在数字人、景点内容、路线决策和运营数据上。

### 6.2 视觉原则

- 主色：松石绿/湖绿色，用于主要操作和当前状态。
- 辅色：佛金/铜金，用于文化强调、演出提醒和重点景点。
- 中性色：白、浅灰、墨色文字为主，避免整站被米色、深蓝或单一绿色统治。
- 字体：中文优先使用系统字体；标题 600 到 700 字重，正文 16px 起。
- 图标：统一使用 lucide 或同一套 SVG 图标，不用 emoji 当结构图标。
- 卡片：只用于景点卡、路线站点、后台指标卡，不做卡片套卡片。
- 圆角：8px 以内，终端大按钮可放宽到 12px。

### 6.3 手机端布局

- 375px 小屏优先。
- 顶部展示景区名、当前状态和简短问候。
- 数字人区域占首屏核心，但不能挤掉输入区。
- 底部固定操作区：语音、文本、拍照、路线四个主入口。
- 聊天消息展示来源卡片和置信度，但默认折叠长引用。
- 拍照识景使用底部弹层，失败时给“重试”和“手动选择景点”。
- 路线页用纵向时间线展示点位、停留时间和讲解重点。

### 6.4 景区终端布局

- 目标分辨率 1920x1080 横屏。
- 左侧或中间为大数字人，右侧为热门问题、路线入口和当前讲解卡。
- 触控目标不小于 64px，高频按钮间距不小于 16px。
- 闲时展示欢迎态、热门问题和“扫码带走路线”提示。
- 所有流程必须能在无键盘情况下完成。
- 超时 60 到 120 秒自动回欢迎态，清空上一位游客状态。

### 6.5 管理后台布局

- 管理后台走密集但清晰的运营工具风格，不做景区宣传页。
- 左侧导航：概览、知识库、数字人、交互日志、行为分析、评测。
- 概览页首屏展示服务人次、满意度、平均延迟、低置信度问题。
- 图表优先使用折线图、柱状图、排行榜、漏斗和表格；不用纯装饰图。
- 所有图表必须有标题、单位、时间范围和空状态。

### 6.6 可访问性与响应式

- 正文对比度不低于 4.5:1。
- 所有可点击元素有可见 focus 状态。
- 触控目标手机不小于 44px，终端不小于 64px。
- 支持 reduced-motion，关闭非必要动画。
- 断网、模型不可用、识别失败、无日志、无知识切片都要有清楚空状态。
- 断点：375、768、1024、1440、1920。

## 9. 安全、隐私与模型兜底

### 9.1 API Key 与配置

- 所有模型 API Key 只放在后端 `.env`，不进入前端包。
- 前端只能请求后端 provider API，不能直接调用模型厂商。
- README 要写清楚无 API Key 时如何启动 mock provider。

### 9.2 公共终端安全

- `/kiosk` 不持久化游客私人状态到 localStorage。
- 终端分享路线使用短期 share_code，不暴露 session_id。
- 超时清空后，上一位游客的问题、路线和图片不可见。

### 9.3 模型兜底

- LLM、Embedding、VLM、TTS 都要有 provider 抽象。
- 至少 LLM 和 VLM 有 mock provider。
- 后台展示 provider 状态：正常、降级、不可用。
- 演示模式可以固定使用 mock/fixture 数据，保证视频录制稳定。

### 9.4 AI 内容与隐私提示

- 游客端和 Kiosk 首屏或底部应展示简短提示：本系统由 AI 数字人提供导览建议，景区事实信息以来源卡片为准。
- 图片识景只用于当前识别和演示日志，不在前端展示原始图片历史。
- 公共终端不保存个人身份信息，不展示上一位游客的私人问题和路线偏好。
- 后台只展示聚合数据、脱敏日志和公开样例行业画像；公开样例数据不得冒充真实景区运营数据。
- 深度合成/AI 生成讲解内容应通过 UI 文案做显著提示，避免游客误认为全部为人工讲解。

从导览工具升级为景区智能服务中台。

|------|----------|----------|------|
| 资料解析 | 解析出灵山 16 个景点、拈花湾 6 个景点 | docx 表格结构变化 | 输出错误行和源文件名，允许手动修正 JSON |
| RAG 问答 | 返回答案、来源、置信度 | 检索为空或模型超时 | 返回低置信度兜底和人工服务建议 |
| 图片识景 | 返回景点 ID 和讲解 | VLM 不可用或识别错 | mock provider、手动确认景点 |
| TTS/语音 | 能播放语音并驱动口型 | 浏览器不支持或播放失败 | 显示文本答案，数字人切换文本讲解态 |
| 路线推荐 | 生成点位顺序和停留时间 | 用户输入过少 | 用默认画像追问或给推荐模板 |
| Kiosk 会话 | 超时清空状态 | 上一位游客信息残留 | 不使用持久化私人状态，分享码过期 |
| 管理大屏 | 展示日志和公开数据洞察 | 无真实日志 | 使用演示样例并明确标注 |
| 评测脚本 | 输出准确率和失败样例 | provider 不可用 | 自动切 mock provider，标记报告模式 |

关键测试类型：

- 单元测试：资料解析、RAG 检索规则、路线模板、share_code 过期。
- API 测试：health、attractions、chat、recognize、routes、feedback。
- UI smoke：手机首页、Kiosk 首页、后台概览。
- 评测：50 题问答、8 到 10 张识景样例、平均延迟。

## 16. 并行开发策略

| 任务线 | 模块 | 依赖 |
|--------|------|------|
| A. 资料与后端基础 | parser、database、attractions API | 无 |
| B. 前端壳与设计系统 | mobile、kiosk、admin shell | 无 |
| C. RAG 与模型 provider | knowledge、chat、provider | A |
| D. 数字人与语音 | digital human、TTS、ASR | B、C |
| E. 路线与识景 | route、recognition、share | A、B、C |
| F. 后台与评测 | dashboard、evals、analytics | A、C |

推荐顺序：

- 先并行 A + B。
- A 完成后启动 C。
- B/C 稳定后并行 D + E + F。
- 最后统一做演示脚本、PPT、视频和部署手册。
